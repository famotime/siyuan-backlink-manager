# 反链上下文 WP-02 重构收口检查

## 文档目的

本文用于在模块重构完成后，确认反链上下文优化相关链路是否已经收口，并判断既有开发计划是否需要调整。

本文只做结构与职责检查，不做代码改动。

## 检查范围

- 查询层：`src/service/backlink/backlink-query-loaders.js`
- 收集层：`src/service/backlink/backlink-panel-data-collectors.js`
- 装配层：`src/service/backlink/backlink-panel-data-assembly.js`
- 渲染数据层：`src/service/backlink/backlink-render-data.js`
- 编排入口：`src/service/backlink/backlink-data.ts`
- 数据模型：`src/models/backlink-model.ts`
- 基础节点构建：`src/service/backlink/backlink-panel-base-data-builder.js`
- 过滤层：`src/service/backlink/backlink-filtering.js`

检查日期：`2026-03-15`

## 结论摘要

结论一：既定优化目标不需要改，开发优先级也不需要改。  
结论二：`WP-02` 仍然必须先做，但要先补“收口闭环”，再做“相邻块接入”。  
结论三：当前重构后的模块分层方向是成立的，但“相邻块”相关链路仍停留在半接入状态，不能直接进入后续 `WP-03`。

## 当前链路检查结果

### 1. 查询层已拆分，但没有相邻块查询能力

当前查询层已经把以下能力拆开：

- 反链块查询
- 标题子级查询
- 列表项子级查询
- 父级查询

但没有看到以下内容：

- 相邻块查询函数
- 相邻块查询参数
- 相邻块 SQL 生成与加载入口

这说明相邻块还没有进入“查询层正式能力”。

### 2. 收集层没有相邻块采集实现

当前收集层已经有：

- `collectBacklinkBlocks`
- `collectHeadlineChildBlocks`
- `collectListItemTreeNodes`
- `collectParentBlocks`

但没有 `collectSiblingBlocks` 的实现或导出。  
这意味着即使上游拿到了相邻块结果，当前收集层也无法把它们正式写入反链节点。

### 3. 装配层保留了相邻块预留接口，但接口未闭环

装配层 `buildBacklinkPanelData` 已经出现以下预留接口：

- `collectSiblingBlocks`
- `paramObj.backlinkSiblingBlockGroupArray`

说明重构时已经为相邻块留出装配位置。  
但由于查询层、收集层、编排层没有同步接入，这里目前仍是悬空扩展点，不是可工作的正式能力。

### 4. 编排入口尚未把相邻块数据链路接通

当前 `backlink-data.ts` 会：

- 查询反链块
- 查询父级块
- 查询标题子级块
- 查询列表项子级块
- 调用装配层生成 `backlinkPanelData`

但没有：

- 查询相邻块
- 传入 `backlinkSiblingBlockGroupArray`
- 注入 `collectSiblingBlocks`

因此，装配层中的相邻块预留入口实际上不会被触发。

### 5. 数据模型尚未给相邻块预留稳定表达

当前 `IBacklinkBlockNode` 和节点初始化结构主要包含：

- 当前块
- 文档块
- `parentMarkdown`
- `headlineChildMarkdown`
- 列表项树 `parentListItemTreeNode`
- 若干定义块 ID 集合

未看到稳定的相邻块字段或通用上下文片段容器。  
这意味着即使补上采集函数，也还没有统一的数据位置去承接相邻块上下文。

### 6. 搜索与筛选仍依赖旧的拼接型上下文

当前关键字匹配范围主要由以下文本拼接得到：

- 当前块
- 当前结果文档信息
- 父级文本
- 标题子级文本
- 列表项子级文本

未包含相邻块。  
这说明“相邻块接入”不是只改查询或采集即可完成，而是必须同时补齐搜索、筛选、展示链路。

### 7. 当前过滤层仍是结构特判，不是统一片段规则

`backlink-filtering.js` 仍然主要围绕：

- 关联定义块 ID 集合
- 列表项树的 include/exclude 特判

来做结果裁剪。  
这进一步说明 `WP-03` 的必要性没有变化，且不能跳过。

## 对开发计划的影响

### 不变部分

- 总体目标不变
- 各阶段优先级不变
- `WP-03` 到 `WP-13` 的依赖关系基本不变

### 需要明确化的部分

`WP-02` 应理解为两个连续子步骤：

1. 重构收口闭环
2. 相邻块上下文正式接入

也就是说，`WP-02` 不是“直接开始补相邻块显示”，而是先把新架构里的接口链路补完整，再把相邻块纳入正式上下文来源。

### 建议的实现前置条件

进入 `WP-02` 编码前，建议先满足以下前置条件：

1. 明确相邻块查询输出结构
2. 明确收集层写入节点的目标字段
3. 明确相邻块在当前兼容模型中的最小落点
4. 明确搜索拼接与后续上下文片段模型之间的过渡方案

如果这四点不先确定，后续很容易把逻辑重新堆回 `backlink-data.ts`，抵消这次重构收益。

### 推荐的下一步顺序

1. 先完成 `WP-02` 的“收口闭环检查结果入计划”
2. 再补查询层与收集层的相邻块接口定义
3. 再补装配层到渲染层的最小兼容链路
4. 最后进入 `WP-03`，把搜索、筛选、展示统一接到上下文模型

## 本次检查结论

当前计划需要更新描述和进度记录，但不需要改目标或改优先级。  
下一步应继续执行 `WP-02`，且先做“接口闭环”，再做“相邻块正式接入”。
