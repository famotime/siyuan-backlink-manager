# 反链上下文优化开发计划

## 文档目的

本文用于把《反链上下文设计建议对照当前实现的优化建议》落实为可执行的开发计划。

使用方式：

- 后续开发按本文的阶段和工作包逐项推进
- 每完成一项，直接更新本文中的状态、实际完成日期、备注
- 若开发中发现范围变化，优先更新本文，再继续开发

本文暂不对应具体代码改动，只定义开发顺序、任务边界、依赖关系、验收标准和进度记录方式。

## 重构影响说明

本计划最初制定时，默认很多实现仍集中在单一入口文件中。当前模块重构已经完成，因此后续执行本计划时，需要按新的模块边界推进，而不是继续按单一文件推进。

当前后端主要模块边界：

- 查询层：`backlink-query-loaders.js`
- 收集层：`backlink-panel-data-collectors.js`
- 装配层：`backlink-panel-data-assembly.js`
- 渲染数据层：`backlink-render-data.js`
- 编排入口：`backlink-data.ts`

当前前端主要模块边界：

- 页面容器：`backlink-filter-panel-page.svelte`
- 结果区：`backlink-results-panel.svelte`
- 控件区：`backlink-filter-panel-controls.svelte`
- 面板控制器：`backlink-panel-controller.js`
- 文档导航：`backlink-document-navigation.js`

因此，后续工作包中的“建议涉及文件”应理解为模块组，而不是单个入口文件。

## 计划目标

本轮优化的总目标不是简单增加更多上下文，而是把当前已有的上下文能力统一成一套：

- 分层的上下文模型
- 可解释的搜索命中机制
- 渐进展开的展示策略
- 一致的搜索、筛选、展示规则
- 可控制的上下文预算

## 范围说明

本计划覆盖：

- 反链上下文的数据模型
- 搜索与筛选所依赖的上下文组织方式
- 反链面板的默认展示层级与展开交互
- 上下文来源标记、命中解释、预算控制
- 默认策略与配置模型调整

本计划暂不覆盖：

- 新增复杂的语义分析能力
- AI 生成摘要
- 大规模 UI 视觉重设计
- 与思源官方接口无关的独立索引系统重建

## 状态定义

- `未开始`：尚未进入开发
- `进行中`：正在开发
- `已完成`：开发、验证完成
- `阻塞`：存在外部依赖或方案未决
- `延期`：本阶段不再按原计划推进

## Phase 1：统一上下文模型

### 目标

建立一套能同时服务于搜索、筛选、展示的“上下文片段”模型。  
这是后续所有改动的基础。如果这一层不统一，后面的命中解释、来源标签、分层展示都会变成补丁式实现。

### 工作包

#### WP-01 设计统一上下文片段数据结构

状态：`已完成`

目标：

- 用显式对象表示上下文，而不是继续依赖拼接字符串和分散字段

建议设计内容：

- `sourceType`：来源类型
- `blockId`：来源块 ID
- `rootId`：来源文档 ID
- `text`：用于搜索与摘要的文本
- `anchorText`：锚文本
- `searchable`：是否参与搜索
- `filterable`：是否参与筛选
- `defaultVisible`：默认是否展示
- `visibilityLevel`：所属展示层级
- `matched`：是否命中
- `matchReason`：命中原因

具体任务：

1. 定义上下文片段接口
2. 定义来源类型枚举
3. 定义展示层级枚举
4. 明确上下文片段与当前 `IBacklinkBlockNode`、`IBacklinkData` 的关系
5. 确定是增量接入还是替换式接入

建议涉及文件：

- `src/models/backlink-model.ts`
- `src/service/backlink/backlink-query-loaders.js`
- `src/service/backlink/backlink-panel-data-collectors.js`
- `src/service/backlink/backlink-panel-data-assembly.js`
- `src/service/backlink/backlink-render-data.js`
- `src/service/backlink/backlink-data.ts`

验收标准：

- 可以用统一对象表达现有各类上下文来源
- 搜索、筛选、展示后续都能引用该结构

## 实施顺序建议

建议严格按以下顺序推进：

1. Phase 0
2. Phase 1
3. Phase 2
4. Phase 3
5. Phase 4
6. Phase 5

不建议跳过 Phase 1 直接做 UI 层改动。  
如果上下文模型不先统一，后续的命中摘要、来源标签、展示层级只会继续依附旧逻辑，成本会越来越高。

另外，`WP-02` 中的“重构收口检查”不建议跳过。  
如果当前模块边界还存在未收口接口，后续功能开发会重新把逻辑拉回到入口文件，抵消这次重构的价值。

## 当前结论

当前最应该先做的，不是继续在现有字段上堆更多判断，而是先完成下面三件事：

1. 统一上下文片段模型
2. 把相邻块纳入正式上下文来源
3. 建立命中解释和分层展示的基础设施

只有这三件事做好，后面的来源标签、预算控制、展示模式切换和默认策略预设才会是低成本、可维护的增量开发。
