# 反链上下文基线与术语表

## 文档目的

本文用于记录当前仓库已经落地的上下文模型基线，并统一术语口径。

正式规则仍以：

- `docs/上下文状态渐进显示规则.md`

为唯一准绳。

本文回答的是：

1. 当前代码里这些术语分别对应什么
2. 当前实现已经对齐了哪些规则
3. 仍然剩下哪些明确 GAP

## 当前术语表

| 术语 | 当前口径 |
| --- | --- |
| 定义块 | 当前文档中被其他块引用的内容块 |
| 反链块 | 引用了定义块的来源块 |
| 焦点块 | 当前这条反链真正落点的块，通常也是 `sourceWindow/contextPlan.identity.focusBlockId` |
| 锚点单元 | 当前层级必须展示的最小原文单元；普通块通常是当前块，列表场景通常是列表项壳与焦点子块 |
| 结构壳 | 为了让用户知道内容属于哪个结构而保留的最小结构块，例如标题行、列表项壳 |
| 相邻单元 | 与锚点文档顺序相邻、且同层级优先的前后语义单元 |
| 所属标题节 | 从最近标题开始，到下一个同级或更高级标题前结束的范围 |
| 上下文层级 | `core / nearby / extended / full` 四层正文范围 |
| 元信息 | 不计入四层正文范围的辅助信息；当前已落到 `contextBundle.metaInfo`，典型是 `documentTitle`、`headingPath`、`listPath` |
| 说明片段 | 用于命中解释、来源说明和预算控制的片段；当前主字段是 `contextBundle.explanationFragments` |
| 正文窗口 | 当前正文渲染使用的原文窗口；当前主字段是 `sourceWindow.contextPlan.bodyRange` |
| 线性可见块 | 当前层级按文档顺序排列的可见块序列；当前主字段是 `orderedVisibleBlockIds` |
| 折叠占位块 | 属于正文窗口但默认不展开的块；当前主字段是 `collapsedBlockIds` |
| 显式降级 | source window 缺失时，界面明确提示“原文上下文不可用，当前显示为降级结果” |

## 当前实现基线

### 1. 正文范围与元信息已分层

当前实现：

- `document` 来源不再作为默认正文片段
- 文档标题、标题路径、列表路径等辅助信息已移到 `contextBundle.metaInfo`
- 预算与层级说明优先基于 `explanationFragments`

代码位置：

- `src/service/backlink/backlink-context.js`
- `src/service/backlink/backlink-context-rules.js`
- `src/service/backlink/backlink-render-data.js`

当前状态：

- 已基本对齐“定位信息与正文范围分开计算”

### 2. 正文窗口已开始统一到 contextPlan

当前实现：

- `sourceWindow.contextPlan.identity/bodyRange/orderedVisibleBlockIds/collapsedBlockIds` 已成为正文消费主骨架
- getter 会合并 `contextPlan` 与 legacy root 字段
- 渲染与 DOM 过滤优先消费 getter 结果
- `startBlockId / endBlockId` 已与真实正文边界对齐，不再跳过标题壳或列表壳

代码位置：

- `src/service/backlink/backlink-source-window.js`
- `src/components/panel/backlink-document-interaction.js`
- `src/components/panel/backlink-protyle-dom.js`
- `src/components/panel/backlink-protyle-rendering.js`

当前状态：

- 已进入“统一 planner 收口”阶段
- legacy root 字段仍然存在，但已明显退到兼容层

### 3. 列表规则已按正式规则收紧

当前实现：

- 列表 `core` 的 `bodyRange` 已收紧到“结构壳 + 焦点块”的最小连续原文范围
- 列表 `nearby` 默认只带邻居壳层与首层直接子项
- `collapsedBlockIds` 已真正驱动列表折叠，深层子树不再默认落入当前层级正文窗口

代码位置：

- `src/service/backlink/backlink-source-window.js`
- `tests/backlink-source-window.test.js`

当前状态：

- 已对齐正式规则中的列表 `core/nearby`

### 4. 正式正文路径已不再依赖 preview assembly

当前实现：

- 生产正文主路径不再 import `backlink-preview-assembly.js`
- `previewSequence` 不再自动进入生产 `contextBundle`
- source window 缺失时会显示显式降级提示，而不是伪装成正式正文

代码位置：

- `src/components/panel/backlink-document-interaction.js`
- `src/service/backlink/backlink-render-data.js`
- `src/components/panel/backlink-panel-header.js`

当前状态：

- 正式正文与异常降级已分离
- preview assembly 仍作为兼容模块保留，但不再是正文主链路

### 5. 同 root 焦点刷新已具备最小闭环

当前实现：

- 数据层不再把 `focusBlockId` 入口硬清空
- 文档底部反链宿主会把当前 protyle 焦点块传入面板
- 同 root 下会更新现有面板 props，而不是直接短路返回
- 控制器在同 root 焦点刷新时复用现有 `queryParams`、面板展开态和文档 active index

代码位置：

- `src/service/backlink/backlink-data.ts`
- `src/service/plugin/backlink-panel-host.js`
- `src/service/plugin/DocumentService.ts`
- `src/components/panel/backlink-panel-controller.js`

当前状态：

- “焦点变化后按当前层级重算”已有输入链路和状态保持基础
- 编辑/拖拽后的当前文档组局部刷新闭环已接通

## 当前剩余兼容边界

### 1. explanation/source 仍保留旧来源桶模型

当前情况：

- `contextBundle.fragments/explanationFragments` 仍服务于命中来源、摘要和预算提示
- `expanded` 仍作为兼容来源存在，但不再驱动正文窗口

影响：

- 说明层和正文层已分层，但数据结构上仍存在兼容字段

### 2. preview assembly 仍作为兼容模块存在

当前情况：

- 正文主链路已经不用它
- 兼容预览模块与测试资产仍保留

影响：

- 架构上仍有历史兼容成本，但不影响反链页签正式正文规则

## 当前结论

当前实现已经从“正文范围、说明来源、渲染路径三套并行”收敛到：

- 正文：`contextPlan + sourceWindow getter`
- 说明：`explanationFragments + metaInfo`
- 状态：文档层级保持 + 同 root 焦点刷新

后续真正需要收尾的重点只剩两类：

1. 继续把 legacy sourceWindow / preview 兼容字段压缩到 adapter 层
2. 进一步收敛 explanation/source 的历史兼容模型
