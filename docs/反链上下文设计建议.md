# 反链上下文设计建议

## 目的

反链要真正有用，关键不只是“知道哪里引用了我”，而是“看见这条引用时，能立刻理解它为什么引用、在什么语境下引用、我接下来要不要点进去处理”。

因此，反链面板的上下文设计不应只围绕技术上的块层级展开，而应围绕用户在阅读反链时最常见的三个问题：

1. 这条反链在说什么
2. 它为什么会引用当前块
3. 这条反链在原文中的作用是什么

如果面板只能看到一小段孤立文本，用户通常还要再打开文档、展开标题、回看上下文，反链的效率会明显下降。

## 设计原则

### 1. 先给定位上下文，再给语义上下文

用户首先需要知道这条反链来自哪里，因此默认应优先展示：

- 文档名
- 面包屑或标题路径
- 当前反链块本身

但仅有定位通常不够，还需要补足“这句话是什么意思”的语义上下文。语义上下文应根据反链所处位置动态决定，而不是一律只看父级或子级。

### 2. 上下文的范围应服从“最小足够理解”

上下文过少，用户看不懂；上下文过多，用户扫不动。

合理的策略不是始终展示更多，而是：

- 默认展示最小但足够理解的一层上下文
- 对天然依赖结构理解的场景，再补第二层上下文
- 更大的范围通过展开操作获得

### 3. 结构位置决定上下文重点

同样是一条反链，出现在句子、列表项、标题、日志记录、索引页时，用户期待的上下文完全不同。

因此应按“反链承担的表达角色”来决定展示内容，而不是只按“块类型”来处理。

### 4. 反链的最佳建立位置，应是“承载语义”的位置

用户建立反链时，最有价值的不是把链接放到任何能放的位置，而是放到“表达判断、决策、结论、任务、证据”的那一行或那一段。

换句话说，反链最好建在“有意义的表述单元”上，而不是建在纯导航、纯罗列、纯标题占位的位置。

## 建议的上下文层级

从用户体验出发，一条反链的上下文可以分成四层：

- 定位层：文档名、标题路径、块所在结构
- 核心层：直接引用当前块的那一行或那一段
- 解释层：帮助理解该引用含义的父级、子级、前后句
- 操作层：时间、任务状态、文档类型、是否为索引页等辅助信息

默认展示建议以“定位层 + 核心层”为基础，再按场景选择性补“解释层”。“操作层”适合弱化显示，但不宜完全缺失。

## 分场景说明

下文重点回答两个问题：

1. 当反链位于不同位置时，面板应展示哪些上下文
2. 用户更适合把反链建立在什么位置

## 场景 1：反链出现在句内或短段落中

### 用户期待

这是最典型、也最容易理解的反链场景。用户通常希望一眼看到：

- 反链所在句子
- 这句话前后一句在说什么
- 这句话属于哪个主题或标题

### 建议展示的上下文

- 文档名
- 标题路径
- 当前句或当前段
- 当前段前后相邻的 1 个语义单元

这里的重点不是“父级块”本身，而是“前后语义是否完整”。如果该句本身就能独立表达，则无需额外展开太多内容；如果它是“因此”“见上”“如下所示”这类依赖前文的句子，就需要补前一句或整段。

### 建议用户建立反链的位置

优先建立在：

- 明确表达判断的句子
- 明确说明关系的句子
- 明确给出结论、原因、证据、对比的句子

不建议建立在：

- 只有一个块引用、没有解释的孤立句子
- “见这里”“相关内容”“参考如下”这类弱语义句子

### 推荐示例

推荐：

```md
本方案放弃轮询，改为事件触发，原因见 ((事件模型设计))。
```

不推荐：

```md
参考：((事件模型设计))
```

## 场景 2：反链是一个独立段落

### 用户期待

当反链本身就是一整段说明时，用户通常希望直接在面板里读完，而不是只看到段落开头。

### 建议展示的上下文

- 文档名
- 标题路径
- 完整段落
- 若段落开头或结尾明显依赖上下文，则补充前一段或后一段的摘要

### 建议用户建立反链的位置

优先把反链建在：

- 结论段
- 方案说明段
- 对比分析段
- 经验总结段

这是高质量反链的理想位置，因为一整段本身就承载了完整语义。

## 场景 3：反链位于列表项的叶子节点

### 用户期待

列表项的叶子节点常用于记录任务、结论、事实、检查项。用户最需要的是知道：

- 这条子项属于哪个列表主题
- 当前子项本身写了什么

### 建议展示的上下文

- 文档名
- 标题路径
- 当前列表项
- 直接父列表项

通常不应默认展示大量兄弟项，因为兄弟项多数会制造噪音。只有当列表是强对比结构时，兄弟项才有价值。

### 建议用户建立反链的位置

优先建立在：

- 有动作含义的待办项
- 有判断含义的结论项
- 有状态含义的检查项

例如：

```md
- 性能优化
  - 首屏延迟主要来自缓存预热，关联 ((启动链路分析))
```

不建议只放在父列表项标题上，除非父项本身就能表达完整意思。

## 场景 4：反链位于列表项的父节点，真正语义在子项里

### 用户期待

这是最容易产生误判的场景。父列表项常常只是一个主题，如“问题分析”“优化方案”“会议结论”，真正有信息量的是下面的子项。

用户在面板里如果只能看到父项标题，通常并不能理解为什么引用了当前块。

### 建议展示的上下文

- 文档名
- 标题路径
- 父列表项本身
- 与该父项直接相关的前几个子项

如果父项下内容很多，默认不应全展开，而应优先展示：

- 包含当前反链的那个子项
- 能解释父项含义的前 1 到 3 个子项

### 建议用户建立反链的位置

更推荐的做法不是把反链建在父项上，而是：

- 直接建在承载具体含义的子项上
- 或在父项下补一条摘要子项，再把反链建在摘要子项上

推荐：

```md
- 问题分析
  - 用户误以为同步完成，原因是界面没有展示处理中状态，关联 ((异步交互规范))
```

不推荐：

```md
- 问题分析 ((异步交互规范))
  - 用户误以为同步完成
  - 界面没有展示处理中状态
```

后者的反链在面板里很容易显得过于空泛。

## 场景 5：反链位于标题本身

### 用户期待

标题型反链的价值很高，也很危险。

价值在于：它通常说明“整个小节都与当前块相关”。  
危险在于：标题本身往往只有主题名，没有具体论述。

用户在看到标题型反链时，真正想知道的是：

- 这个标题下到底讲了什么
- 该标题与当前块是“主题相关”还是“明确引用”

### 建议展示的上下文

- 文档名
- 完整标题路径
- 当前标题
- 标题下第一段正文或前几个子块

如果标题下没有正文，只有一个空标题或若干无解释列表项，那么应降低这类反链的信息权重。

### 建议用户建立反链的位置

只有在以下情况才建议把反链放在标题上：

- 该节整体都围绕当前块展开
- 标题本身已经足够具体
- 标题下的第一段能马上解释关联关系

推荐：

```md
## 为什么放弃轮询并采用事件驱动 ((事件模型设计))
```

不推荐：

```md
## 相关内容 ((事件模型设计))
```

如果标题只是占位，最好把反链放到标题下的摘要段，而不是标题行本身。

## 场景 6：反链位于标题下的正文块

### 用户期待

这是结构化笔记中非常常见的场景。用户通常希望：

- 知道这段正文属于哪个标题
- 看到这段正文本身
- 在必要时了解同段或相邻段落

### 建议展示的上下文

- 文档名
- 标题路径
- 当前正文块
- 若当前块过短，则补同节前后相邻 1 个块

相比“标题型反链”，这个场景的核心应始终是正文块本身，因为真正承担语义的是正文，而不是标题标签。

### 建议用户建立反链的位置

优先建立在标题下的摘要段、结论段、方法段，而不是为了省事直接挂在标题上。

## 场景 7：反链位于文档开头、摘要、结论区

### 用户期待

如果反链出现在文档开头摘要、结论总结、TL;DR 之类位置，用户往往会把它理解为“这篇文档整体与当前块相关”。

这类反链通常信息密度高，但也容易过于概括。

### 建议展示的上下文

- 文档名
- 文档类型或摘要标签
- 当前摘要块或总结块
- 可选展示文档创建时间、更新时间

### 建议用户建立反链的位置

适合建立在：

- 一句话总结
- 方案结论
- 文档摘要

但不建议只在文档开头机械堆积多个链接而不写说明。摘要区的反链应承担“概括关系”的作用，而不是替代正文分析。

## 场景 8：反链位于日记、会议记录、项目日志中

### 用户期待

在日志型文档里，时间和情境与内容同样重要。用户看到反链时，不仅关心“写了什么”，还关心：

- 是哪一天或哪次会议提到的
- 它属于哪个议题
- 这是结论、讨论、还是行动项

### 建议展示的上下文

- 文档名
- 日期或会议标题
- 所在议题标题
- 当前反链块
- 若为行动项，补状态或负责人

### 建议用户建立反链的位置

优先建立在：

- 会议结论
- 决策记录
- 待办动作
- 风险或问题记录

不建议建立在纯时间标记、签到信息、流程性记录上，因为这些内容在回看反链时辨识度很低。

## 场景 9：反链位于索引页、导航页、MOC、资料汇总页

### 用户期待

这类页面的反链主要承担“导航价值”，而不是“论述价值”。

用户看到这类反链时，通常希望知道：

- 这是在哪个索引页出现的
- 它在这个索引页里属于哪个分类

但通常不会指望它直接解释“为什么引用”。

### 建议展示的上下文

- 文档名
- 分类标题或父项
- 当前索引项

这类场景不需要展示太多正文，因为正文通常没有太多正文。

### 建议用户建立反链的位置

可以建立，但应明确把它视作“导航型反链”。

更好的做法是：

- 在索引页中保留链接，用于检索和跳转
- 在分析页、项目页、会议页中再建立“语义型反链”

这样用户回看反链时，既能看到入口，也能看到真正的解释内容。

## 场景 10：反链位于表格、属性型结构、字段型记录中

### 用户期待

当反链出现在表格单元格或字段记录中时，孤立展示一个字段值通常意义不大。用户需要知道：

- 它属于哪一行记录
- 是哪个字段引用了当前块
- 这行记录的主标题是什么

### 建议展示的上下文

- 文档名
- 表格名称或记录集合名称
- 当前记录的主键字段或标题字段
- 当前单元格内容
- 字段名

### 建议用户建立反链的位置

优先建立在：

- 备注字段
- 结论字段
- 说明字段
- 关联原因字段

不建议主要依赖：

- 编号字段
- 标签字段
- 纯枚举字段

因为这些位置虽然能形成反链，但对回看者的解释价值很弱。

## 场景 11：一条反链同时引用多个定义块

### 用户期待

用户这时最想知道的，不只是“它还引用了谁”，而是：

- 这些引用是并列关系、因果关系，还是比较关系
- 当前块在这组引用中的角色是什么

### 建议展示的上下文

- 文档名
- 标题路径
- 当前反链块完整内容
- 关联的定义块列表
- 若可能，突出当前块在句中的位置

### 建议用户建立反链的位置

优先把多链接放在能体现关系的句子里，例如：

- A 依赖 B
- A 对比 B
- A 继承自 B
- A 与 B 一起构成 C

不要只写一串链接列表，否则虽有反链，但缺乏可解释性。

## 场景 12：反链位于模板、固定栏目、重复性样板内容中

### 用户期待

用户通常不希望样板文字淹没真正重要的反链。

例如每天都会自动生成“相关项目：((X))”的模板行，这类反链会大量出现，但信息含量很低。

### 建议展示的上下文

- 文档名
- 模板栏目名
- 当前样板行

但从排序和感知上，这类反链应弱化处理，因为用户真正需要优先看到的是有新信息的反链。

### 建议用户建立反链的位置

模板中可以保留导航链接，但更建议用户在当天真正的记录、结论、任务或复盘段落中再建立一条有语义的反链。

## 不同场景下的默认展示建议

为了兼顾信息完整性和浏览效率，可以采用以下默认策略：

| 场景 | 默认应展示 | 默认不必展示 | 建议按需展开 |
| --- | --- | --- | --- |
| 句内/短段落 | 文档名、标题路径、当前句或当前段 | 大量父级和兄弟项 | 前后句、整段 |
| 独立段落 | 文档名、标题路径、完整段落 | 过多相邻段 | 前后段 |
| 列表叶子项 | 文档名、标题路径、当前项、直接父项 | 全部兄弟项 | 子项详情、相关兄弟项 |
| 列表父项 | 文档名、标题路径、父项、关键子项 | 全量子树 | 子树展开 |
| 标题本身 | 文档名、完整标题路径、标题、首段正文 | 整节全部内容 | 本节更多块 |
| 标题下正文 | 文档名、标题路径、当前正文块 | 整节全部兄弟块 | 相邻块 |
| 日志/会议 | 文档名、日期或会议名、议题、当前块 | 无关流程记录 | 状态、负责人、前后项 |
| 索引/MOC | 文档名、分类路径、当前索引项 | 大段正文 | 所属分类更多条目 |
| 表格/字段 | 文档名、记录标题、字段名、当前值 | 整表所有列 | 该记录关键字段 |

## 建立高质量反链的通用建议

### 建议 1：把反链建在“有判断的表达单元”上

最好的反链位置通常是：

- 结论
- 决策
- 原因
- 证据
- 对比
- 动作
- 风险

这些位置回看时，用户最容易理解“为什么引用了这个块”。

### 建议 2：避免把反链建在纯占位结构上

尽量避免主要建立在：

- 空标题
- 泛化标题
- 纯导航列表
- 模板样板行
- 只有链接没有说明的孤立项

这些位置虽然能形成反链，但阅读价值低。

### 建议 3：如果必须建在标题或父项上，应补摘要

当结构层级必须保留时，更好的方式是：

- 标题下补一段摘要
- 父列表项下补一条说明子项
- 在摘要或说明块上建立反链

这样既保留结构，也提高反链可读性。

### 建议 4：一条反链最好只承载一个清晰意图

如果一个块里既是导航、又是总结、又是任务集合、又塞入很多链接，回看时会非常难以筛选。

更好的做法是按意图拆分：

- 导航归导航
- 分析归分析
- 决策归决策
- 任务归任务

### 建议 5：让反链在离开原文后仍能成立

判断一个反链位置好不好，可以用一个简单标准：

如果把这条反链单独放进反链面板，用户还能大致看懂它在表达什么，那这个位置就基本合格。

如果离开原文后就只剩下“相关内容”“见这里”“方案如下”，那说明建立位置太弱。

## 推荐的用户实践

如果用户希望自己的反链在长期使用中保持高价值，建议形成以下习惯：

1. 在结论句、决策句、行动句上建反链
2. 在标题或父级结构中补一句摘要，再把反链建在摘要上
3. 导航页保留链接，但不要指望导航页反链承担解释任务
4. 日志和会议记录中的反链，尽量落在“结论”和“行动项”而不是流水记录上
5. 多个链接同时出现时，用自然语言写清它们之间的关系

## 一句话总结

反链面板不应只展示“引用发生在哪个块”，而应展示“这条引用在原文中承担什么语义角色”。  
从用户体验出发，最合理的策略是按反链所在位置动态补足最小但足够理解的上下文；从用户使用习惯出发，最值得鼓励的是把反链建立在结论、判断、动作和证据这些真正承载意义的位置上。
