# 反链上下文设计建议对照当前实现的优化建议

> 注意
>
> 本文是面向设计思路的对照分析文档，不是当前正式规则定义。
>
> 其中若出现以下旧表述：
>
> - `扩展` 约等于最近标题区间
> - 列表默认展示整列表子树
> - `nearby` 直接带出完整邻居父项内容
>
> 均应以 `docs/上下文状态渐进显示规则.md` 的正式定义为准。

## 目的

本文不是对具体语义场景逐条复述，而是从通用规则出发，对照当前代码实现，分析反链面板在“上下文呈现”方面还可以如何优化。

目标是回答三个问题：

1. 当前实现已经具备了哪些通用能力
2. 当前实现与理想的上下文设计规则相比，差异主要在哪里
3. 如果暂不改代码，后续应优先考虑哪些优化方向

## 对照范围

本次对照主要参考以下内容：

- 设计建议文档：`docs/反链上下文设计建议.md`

- 当前插件设置项：`src/models/setting-model.ts`、`src/service/setting/SettingService.ts`

## 当前实现的通用能力

从代码上看，当前实现已经具备几项很重要的基础能力。

### 1. 已经不是“只看反链块本身”

当前实现并不是只拿反链块正文做处理，而是会收集多个结构来源的文本：

- 反链块自身
- 反链所在文档信息
- 父级路径文本
- 标题下子级文本
- 列表项下子级文本

这说明当前实现已经具备“结构化上下文”的基础，而不是纯文本命中。

### 2. 已经支持按结构来源扩大关联范围

当前设置中已有这些通用开关：

- 查询父级关联定义块
- 查询列表项下关联定义块
- 查询标题下关联定义块

这说明当前实现已经意识到：反链的理解范围不一定只在当前块。

### 3. 已经提供了从局部到全文的兜底路径

当前渲染不是直接把整篇文档展开，而是默认显示裁剪后的文档片段，并允许用户切换到全文模式。

这说明当前实现已经具备“先局部、再展开”的基本交互思路。

### 4. 已经考虑了列表和标题的可折叠性

当前实现对列表项和标题展开状态有保存和恢复处理，也支持默认展开到某一层级。这对长文档场景很关键，说明实现已经开始处理“上下文过多”的问题。

## 与通用设计规则的主要差异

下面按“通用规则”来对照，而不按语义场景来拆。

## 规则 1：搜索范围和展示范围应尽量一致，至少应可解释

### 设计原则

如果一个结果被搜索命中，用户应能快速看见“它为什么命中”。  
即使搜索范围大于默认展示范围，也应让命中来源是可解释的。

### 当前实现

当前关键字匹配会把这些内容拼接起来统一匹配：

- 当前反链块
- 文档信息
- 父级文本
- 标题下子级文本
- 列表项下子级文本

但默认展示时，并不会显式告诉用户：

- 这条结果是因为父级命中
- 还是因为子级命中
- 还是因为文档名命中

全文展开可以兜底，但在默认视图下，命中来源并不总是直接可见。

### 优化建议

- 建立“命中来源可解释”规则
- 关键字命中后，结果项应能标出命中层级，例如：`块本身`、`父级`、`子级`、`文档名`
- 如果默认展示中没有把命中的内容露出来，应至少提供一条命中摘要，而不是只高亮当前可见区域
- 搜索范围与默认显示范围可以不同，但这种差异必须对用户透明

### 优先级

高。因为这直接影响用户对搜索结果的信任感。

## 规则 2：上下文应有明确层级，而不是只有“是否纳入”

### 设计原则

好的上下文系统不是简单地把更多文本纳入，而是有明确层级：

- 基础层：最小可理解单元
- 扩展层：补足局部结构
- 全文层：兜底查看

### 当前实现

当前主要通过几个布尔开关控制是否查询父级、列表子级、标题子级。  
这是一种“是否纳入”的实现思路，但还不是“分层上下文”的思路。

当前缺少的不是更多来源，而是：

- 哪些内容属于基础层
- 哪些内容属于扩展层
- 默认应该先给哪一层
- 展开时应该先补哪一层

### 优化建议

- 把上下文来源改为分层模型，而不是单纯的布尔模型
- 建议至少定义三层：
  - 核心上下文：反链块本身 + 基本定位信息
  - 近邻上下文：直接父级、直接子级、相邻块
  - 扩展上下文：整节、整列表子树、全文
- 设置项应优先表达“默认层级”，其次才是某类结构是否启用
- 比起让用户理解 3 个结构开关，更有效的是让用户理解“紧凑 / 平衡 / 展开”这类上下文策略

### 优先级

高。因为这是后续所有展示策略统一化的前提。

## 规则 3：局部展开应是渐进式的，不应只有“局部”和“全文”两档

### 设计原则

上下文查看最理想的过程通常是：

1. 先看最小必要上下文
2. 再看局部扩展
3. 最后再决定是否看全文

### 当前实现

当前的主要跳转是：

- 默认裁剪视图
- 点击后全文视图

中间缺一个“局部扩展层”。

这会带来一个问题：

- 默认视图不够时，用户只能直接切到全文
- 全文一旦展开，信息量又可能过大

### 优化建议

- 在交互上增加中间层，而不是直接二选一
- 建议形成三档：
  - 当前反链
  - 当前反链 + 邻近结构
  - 全文
- 列表和标题可以共用这套规则，不需要各自走完全不同的逻辑

### 优先级

高。因为这能明显降低“看不懂”和“看太多”之间的跳变。

## 规则 4：上下文来源应尽量对称，避免只对某一类结构做特殊处理

### 设计原则

从用户角度看，父级、子级、相邻块、文档信息都只是“上下文来源”。  
它们可以有优先级差异，但不应形成过于割裂的规则。

### 当前实现

当前对不同来源的处理方式不完全对称：

- 父级、标题子级、列表子级都能进入匹配范围
- 但显示时的裁剪和过滤主要对列表项做了更细的处理
- 标题子级和其他结构并没有形成同样清晰的显示裁剪规则

这会导致用户对“为什么某些结构被展开、某些结构只参与搜索、某些结构又只在全文里看见”缺少稳定预期。

### 优化建议

- 把所有结构来源统一抽象成“上下文来源”
- 每种来源都回答同样四个问题：
  - 是否参与搜索
  - 是否默认展示
  - 是否可按需展开
  - 是否能作为筛选依据
- 在规则层保持对称，然后再针对个别结构做轻微差异化

### 优先级

中高。因为它直接影响系统的一致性和可预测性。

## 规则 5：反链展示单元应尽量贴近用户的比较单元

### 设计原则

用户在看反链时，真正比较的是“反链上下文单元”，而不是一定在比较“文档壳”。

### 当前实现

当前展示层面有一个明显特点：

- 数据先按反链筛出
- 但展示时会按文档分组
- 每个文档默认只显示一个当前反链，并用左右按钮切换同文档中的其他反链

这个做法有优点：

- 降低同一文档反复出现的噪音
- 便于把同文档反链收束到一个来源

但也有代价：

- 用户默认看到的不是一个“完整的反链结果列表”，而是“文档组中的一个激活项”
- 如果同一文档里有多条高价值反链，它们之间需要额外切换才能比较

### 优化建议

- 将“展示单元”从固定文档分组，改为可切换策略
- 至少考虑两种模式：
  - 按反链上下文平铺
  - 按文档聚合
- 即使保留文档聚合，也建议让用户更清楚当前看到的是“该文档中的第几条反链”，并能快速展开同文档全部命中项

### 优先级

中高。因为这会直接影响结果浏览效率。

## 规则 6：相邻内容是通用上下文来源，不能只依赖父子结构

### 设计原则

用户理解一个块时，最常依赖的上下文之一是“前后相邻内容”。  
这是一种与标题、列表无关的通用规则。

### 当前实现

当前实现重点覆盖的是：

- 父级路径
- 标题下子级
- 列表项下子级

但没有把“前一个块 / 后一个块 / 相邻段”作为显式上下文来源。

这意味着当前模型更偏结构树，而相对忽略线性阅读上下文。

### 优化建议

- 把“相邻块”加入正式的上下文来源集合
- 在优先级上，相邻内容应当高于大范围子级内容
- 尤其在非列表、非标题场景里，相邻块通常比更深层结构更经济

### 优先级

高。因为这是最通用、最稳定的上下文补足规则之一。

## 规则 7：上下文系统应有明确的预算控制

### 设计原则

上下文不是越多越好，而应遵守预算：

- 最多显示多少个块
- 最多展开多少层
- 最多展示多少字符

### 当前实现

当前实现已有部分预算控制思路，例如：

- 页大小
- 列表展开层级
- 默认裁剪 vs 全文

但上下文本身的预算仍然比较隐式，没有明确表达为一套统一规则。

### 优化建议

- 建立统一的上下文预算模型
- 可以按以下维度控制：
  - 最大块数
  - 最大字符数
  - 最大结构深度
  - 单条结果允许展开的最大子树规模
- 预算应当先作用于近邻上下文，再决定是否继续补父级或子级

### 优先级

中高。因为这决定系统是否能长期保持“可扫读”。

## 规则 8：筛选、匹配、展示最好共享同一套上下文模型

### 设计原则

如果筛选、搜索、展示三者使用不同的上下文定义，用户会觉得系统“会算但不会讲”。

### 当前实现

当前实现里：

- 搜索匹配使用的是拼接后的文本串
- 关联定义块筛选使用的是定义块 ID 集合
- 展示依赖的是 Protyle 对文档片段的渲染

它们彼此有关联，但还不是同一套显式上下文模型。

### 优化建议

- 让搜索、筛选、展示都围绕统一的“上下文片段”对象运转
- 每个片段至少应有：
  - 来源类型
  - 所属块 ID
  - 是否默认显示
  - 是否参与搜索
  - 是否参与筛选
  - 命中状态
- 一旦有了统一上下文片段，命中解释、展示裁剪、按来源过滤都会更自然

### 优先级

高。因为这是从功能叠加走向系统化设计的关键一步。

## 规则 9：上下文命中应可见，而不是只高亮当前能看到的文本

### 设计原则

高亮只是命中的一种表现方式，不等于命中解释。  
如果命中的文本当前不在默认视图里，仅靠高亮是不够的。

### 当前实现

当前会对文档标题元素和 Protyle 内容做关键词高亮。  
但如果匹配点在隐藏的父级、隐藏的子级或未展开区域中，用户不一定能看出为什么命中。

### 优化建议

- 除了高亮，还应有“命中摘要”
- 命中摘要应优先展示：
  - 命中字段
  - 命中片段
  - 命中位于何处
- 高亮应服务于“看见命中”，摘要应服务于“理解命中”

### 优先级

高。因为这直接决定搜索体验是否顺畅。

## 规则 10：默认值应优先服务于“低噪音的可靠理解”

### 设计原则

默认配置的目标，不应是尽量多查，而应是尽量让多数用户第一次看到的结果就足够可靠、又不过载。

### 当前实现

当前默认值是：

- 查询父级关联定义块：开启
- 查询列表项下关联定义块：开启
- 查询标题下关联定义块：关闭

这组默认值能覆盖很多结构，但它反映的是实现可用性优先，不完全等于用户理解效率优先。

### 优化建议

- 默认策略应围绕“最小足够理解”来设定，而不是围绕“能查到更多”
- 更好的方式不是简单改默认开关，而是提供默认策略预设，例如：
  - 紧凑：只看核心上下文
  - 平衡：核心 + 近邻
  - 扩展：核心 + 近邻 + 结构扩展
- 让默认体验先稳定，再让高级用户继续放大范围

### 优先级

中高。因为默认值会显著影响用户对整个插件的第一印象。

## 规则 11：上下文来源不应被重复计入或隐式加权

### 设计原则

如果同一来源被重复纳入，会导致匹配、排序、理解都偏向某个来源，削弱结果的可预测性。

### 当前实现

当前搜索拼接时，文档块信息存在重复拼接现象：

- `selfDocumendMarkdown`
- `docContent`

二者来自同一个 `documentBlock`

这虽然不一定导致明显错误，但会让文档级信息在匹配中被隐式放大。

### 优化建议

- 同一来源应只计入一次
- 如果不同来源有轻重，应显式设计权重，而不是通过重复拼接形成隐式权重
- 后续若引入排序打分，也应按来源类型加权，而不是简单文本累加

### 优先级

中。它更偏系统质量问题，但长期会影响匹配的直觉性。

## 规则 12：上下文来源应对用户可见，而不是只存在于内部逻辑中

### 设计原则

一个好的上下文系统，不仅内部知道“这是父级内容、这是子级内容”，用户也应该知道。

### 当前实现

当前“父级 / 列表子级 / 标题子级”主要存在于内部逻辑和设置项中，结果展示层并没有稳定地把这些来源显式标注出来。

### 优化建议

- 在展示层引入轻量来源标记，例如：
  - `父级`
  - `子级`
  - `相邻`
  - `文档`
- 这些标记不一定要一直显眼，但应当在用户需要解释命中时可见

### 优先级

中高。因为这会显著提升用户对规则的掌控感。

## 建议的优化优先级

如果后续要逐步优化，建议按下面顺序推进。

### 第一优先级

- 统一“搜索命中为什么发生”的解释机制
- 建立“核心 / 近邻 / 扩展 / 全文”的上下文层级
- 增加中间层，而不是只有默认视图和全文视图
- 将相邻块纳入正式上下文来源
- 让搜索、筛选、展示共享统一的上下文模型

### 第二优先级

- 统一各类结构来源的规则表达
- 让展示单元支持“按反链”与“按文档”切换
- 引入来源标签和命中摘要
- 建立统一的上下文预算控制

### 第三优先级

- 调整默认值策略
- 去除重复来源拼接
- 为不同来源显式设计轻量权重，而不是让权重隐式产生

## 可以保留的当前实现思路

虽然当前实现和理想规则还有差距，但有几项思路很值得保留：

- 继续保留结构化上下文，而不是退回纯文本搜索
- 继续保留局部视图到全文视图的兜底路径
- 继续保留列表折叠和标题展开的状态记忆
- 继续保留关联定义块筛选，因为它是高价值的结构筛选能力

问题不在于这些方向错了，而在于它们还没有统一成一套对用户可解释、可预测、可控的上下文规则。

## 一句话总结

当前实现已经具备了结构化上下文、关联块筛选、局部裁剪和全文兜底这些重要基础能力；真正需要优化的，不是继续机械扩充上下文来源，而是把现有来源统一成一套分层、可解释、可预算、可渐进展开的通用上下文规则。
