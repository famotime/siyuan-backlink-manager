# 反链相邻段落显示缺口开发计划

## 文档目的

本文用于把当前问题整理为可执行开发计划：

- 为什么反链面板在 `核心 / 近邻 / 扩展 / 全文` 四种状态切换时，看不到预期的前后段落
- 当前实现链路中，问题具体落在哪一层
- 后续应如何分阶段补齐“局部预览按上下文层级真实变化”的能力

本文聚焦通用规则，不依赖具体语义场景。

## 当前状态

状态：`已完成`

已落地结果：

- 非全文模式已改为按 `visibleFragments` 组装局部预览
- `nearby` 会优先显示前相邻块 / 当前块 / 后相邻块
- `extended` 会在 `nearby` 基础上加入父级与子级结构片段
- `full` 继续保持全文渲染，不与局部预览组装耦合
- 已补齐预览组装、渲染接入与全文回归测试

## 当前现象

用户在反链面板中切换四种状态时：

- 能看到状态标签变化
- 能看到局部展开程度有时略有变化
- 但通常看不到明确的“前相邻块 / 后相邻块”正文内容

因此用户的直观感受是：

- 四层状态切换已经存在
- 但真正可感知的上下文内容变化不明显

## 当前实现结论

### 1. 相邻块数据已被采集

当前实现已经具备相邻块采集能力：

- 查询层会按同一 `parent_id` 收集前后直接相邻块
- 收集层会把相邻块内容写入 `previousSiblingMarkdown` / `nextSiblingMarkdown`
- 上下文模型层会生成 `sibling_prev` / `sibling_next` 片段
- 可见性规则层已将二者归入 `nearby`

这说明问题不在“是否有相邻块数据”。

### 2. 四层状态也已生效

当前实现已经具备：

- 文档级 `core / nearby / extended / full` 状态管理
- 标题点击与左右箭头的状态切换
- `visibleFragments` 的层级过滤

这说明问题也不在“状态是否真的切换”。

### 3. 主要缺口在渲染链路

当前 `nearby / extended` 的主要行为是：

- 继续使用同一条 `activeBacklink`
- 让 Protyle 渲染该条反链对应的局部预览
- 再通过展开列表层级、展开标题 more、隐藏部分节点等方式调整显示

但当前没有做到：

- 根据 `contextBundle.visibleFragments` 重新组装局部预览内容
- 把 `sibling_prev` / `sibling_next` 等片段显式拼接进预览
- 让 `core / nearby / extended` 对应不同的预览内容结构

因此当前四层切换更接近：

- `core`：默认局部预览
- `nearby`：局部结构展开更多
- `extended`：局部结构进一步展开
- `full`：切全文

而不是：

- `nearby`：把前后相邻块真实加入预览
- `extended`：把更大结构范围真实加入预览

## 根因拆解

### 根因 1：可见片段只停留在数据层

`visibleFragments` 当前主要用于：

- 过滤与预算控制
- 命中解释
- 来源标签和摘要

但没有进入“正文预览拼装”。

### 根因 2：局部预览仍依赖旧的单条反链渲染模式

当前非全文模式仍把 `activeBacklink` 直接交给 Protyle 渲染。

这意味着：

- 预览内容主体仍由原反链 DOM 决定
- 上下文片段即使已存在，也不会自然出现在正文中

### 根因 3：`nearby / extended` 只控制结构展开，不控制内容注入

当前中间层主要影响：

- 列表展开深度
- 标题 more 展开
- 节点隐藏策略

但不控制：

- 哪些上下文片段需要被新增到正文
- 这些片段按什么顺序显示
- 这些片段如何与当前反链块形成连续阅读体验

### 根因 4：相邻块定义本身较严格

当前 sibling 规则是：

- 同一父块下
- 排序后直接前一个
- 排序后直接后一个

因此即使后续渲染链路补齐，也仍会有一部分反链场景不显示前后段落。这属于规则边界，不属于 bug。

## 开发目标

本轮补齐的目标应明确为：

1. 让 `nearby` 首次切换时，用户能稳定看到“当前块前后相邻内容”的可见变化
2. 让 `extended` 相比 `nearby` 有进一步可感知的正文范围增加
3. 让 `core / nearby / extended / full` 四层在正文层面真正对应四种不同的预览结果
4. 保持现有搜索、筛选、命中说明、预算控制与折叠状态的兼容

## 非目标

本计划暂不包含：

- 改写 sibling 的基础定义
- 引入语义级段落合并
- 引入 AI 摘要或语义补全
- 推翻现有文档分组模式
- 重做整套 UI 视觉设计

## 设计原则

### 1. 数据层与渲染层职责明确分离

数据层负责：

- 收集上下文片段
- 标记层级
- 预算裁剪
- 输出 `visibleFragments`

渲染层负责：

- 将 `visibleFragments` 转为可读预览内容
- 保证各层级预览结果稳定、可预期

### 2. 优先保证 `nearby` 可感知

用户最强烈的预期是：

- 第一次展开就能看到前后段落

因此应优先补齐 `nearby`，而不是先做复杂的 `extended` 结构。

### 3. 先保证层级差异真实存在，再优化提示文案

如果正文内容本身没有变化，再多的状态标签和按钮都无法解决体验问题。

### 4. 维持全文模式作为兜底

`full` 继续保持全文渲染，不与局部预览拼装逻辑耦合。

## 总体方案

建议把非全文模式从“直接复用旧反链局部预览”升级为“按 visibleFragments 构造局部上下文预览”。

推荐的目标行为：

- `core`：仅显示核心命中块
- `nearby`：显示前相邻块 + 当前块 + 后相邻块，并保留必要父级或最近结构线索
- `extended`：在 `nearby` 基础上加入更多父级/子级结构
- `full`：直接全文

这意味着需要新增一层“局部预览组装器”，负责把 `visibleFragments` 变成可渲染结果。

## 分阶段开发计划

## Phase A：现状基线与渲染契约冻结

目标：

- 明确当前局部预览输入输出
- 冻结“局部预览组装器”的数据契约

### WP-SD-01：补齐现状基线测试

优先级：`P0`

状态：`已完成`

任务：

1. 增加测试，证明当前 `nearby` 虽包含 sibling fragment，但正文预览未展示 sibling
2. 增加测试，区分“状态切换成功”和“预览内容未变化”
3. 增加测试，固定 `full` 仍走全文渲染

建议涉及模块：

- `tests/backlink-protyle-rendering.test.js`
- `tests/backlink-document-interaction.test.js`
- 必要时增加新的预览组装测试文件

验收标准：

- 测试能稳定复现当前差距
- 后续改造时可以明确判断问题是否真正解决

### WP-SD-02：定义局部预览组装契约

优先级：`P0`

状态：`已完成`

任务：

1. 定义局部预览的输入结构
2. 定义片段排序规则
3. 定义片段到预览块的映射规则
4. 明确 `core / nearby / extended` 的输出差异

建议输出：

- 预览片段结构定义
- 预览组装顺序规则
- 空片段与重复片段处理规则

建议涉及模块：

- 新增 `src/service/backlink/backlink-preview-assembly.js`
- 对应测试文件

验收标准：

- 后续渲染层不再直接猜测如何使用 `visibleFragments`

## Phase B：让 nearby 真正显示前后相邻块

目标：

- 先解决用户最强感知需求

### WP-SD-03：实现 nearby 预览组装

优先级：`P0`

状态：`已完成`

任务：

1. 从 `visibleFragments` 中抽取 `self / sibling_prev / sibling_next`
2. 组装成稳定顺序的局部预览
3. 保证没有 sibling 时回退到仅显示核心块
4. 去重，避免当前块内容与相邻块重复拼接

建议规则：

- 顺序优先为：前相邻块 -> 当前块 -> 后相邻块
- 若只存在一侧 sibling，仅显示存在的一侧
- 若 sibling 为空内容，则跳过

测试要求：

- 存在前后相邻块时的组装测试
- 仅前/仅后相邻块时的组装测试
- 无相邻块时的回退测试
- 去重测试

验收标准：

- 用户切到 `nearby` 时，正文可明确看到相邻块内容进入预览

### WP-SD-04：将 nearby 预览接入渲染链路

优先级：`P0`

状态：`已完成`

任务：

1. 在非全文模式下改为使用组装后的局部预览数据
2. 保持现有文档级状态切换与反链切换逻辑不变
3. 保持 `Ctrl + 单击`、折叠、翻页等交互兼容

建议涉及模块：

- `src/components/panel/backlink-document-interaction.js`
- `src/components/panel/backlink-protyle-rendering.js`
- `src/components/panel/backlink-panel-controller.js`
- `src/service/backlink/backlink-render-data.js`

测试要求：

- `nearby` 接入后正文内容变化测试
- 文档内切换不同反链时预览同步测试
- 回归测试：`full` 模式不受影响

验收标准：

- `nearby` 状态不再只是“展开多一点”，而是真实改变预览内容

## Phase C：补齐 extended 的结构增量

目标：

- 让 `extended` 相比 `nearby` 有稳定、可理解的增量

### WP-SD-05：定义 extended 增量规则

优先级：`P1`

状态：`已完成`

任务：

1. 明确 `extended` 比 `nearby` 额外加入哪些片段
2. 统一 parent / child_headline / child_list 的拼装顺序
3. 限定最大结构复杂度，避免局部预览失控

建议规则：

- `extended` 在 `nearby` 基础上增加父级线索与局部子级
- 优先保证阅读顺序，而不是把所有片段平铺

测试要求：

- `extended` 相比 `nearby` 的差异测试
- 父级/子级混合时的排序测试
- 复杂结构下的稳定输出测试

验收标准：

- `extended` 呈现出比 `nearby` 更完整，但仍显著小于全文的结果

### WP-SD-06：对接预算与裁剪反馈

优先级：`P1`

状态：`已完成`

任务：

1. 保证局部预览组装后仍服从预算规则
2. 区分“没有更多上下文”和“有上下文但被裁剪”
3. 为后续 UI 提示保留明确状态字段

测试要求：

- 预算命中时的裁剪稳定性测试
- `nearby` / `extended` 被裁剪时的输出测试
- 无上下文与被裁剪两类场景的区分测试

验收标准：

- 用户不会把裁剪误判为“没有前后段落”

## Phase D：渲染兼容与交互收尾

目标：

- 让局部预览新方案与现有交互、折叠、导航、全文模式稳定共存

### WP-SD-07：兼容折叠与节点隐藏逻辑

优先级：`P1`

状态：`已完成`

任务：

1. 审查 `hideOtherListItemElement` 与新预览内容的兼容性
2. 审查列表折叠状态在组装预览模式下是否仍成立
3. 避免新预览内容被旧隐藏逻辑误删

测试要求：

- 列表折叠回显测试
- 标题 more 展开兼容测试
- sibling 预览不被误隐藏测试

验收标准：

- 用户能稳定看到新增上下文，而不会因旧折叠逻辑再次消失

### WP-SD-08：回归测试与文档更新

优先级：`P1`

状态：`已完成`

任务：

1. 完成全量测试
2. 更新相关说明文档与开发计划
3. 记录当前行为边界和已知限制

验收标准：

- 文档、实现、测试三者一致

## 推荐开发顺序

建议严格按以下顺序推进：

1. `WP-SD-01` 固定当前缺口
2. `WP-SD-02` 冻结预览组装契约
3. `WP-SD-03` 实现 `nearby` 组装
4. `WP-SD-04` 接入渲染链路
5. `WP-SD-05` 扩展到 `extended`
6. `WP-SD-06` 接预算与裁剪反馈
7. `WP-SD-07` 清理兼容问题
8. `WP-SD-08` 做全量回归与文档收口

## 每步执行约束

后续若按此计划继续开发，建议保持以下约束：

- 每个工作包先补测试，再写实现
- 每个工作包完成后单独验证
- 每个工作包验证通过后单独提交
- 若发现预览组装契约需要调整，先更新本文再改代码

## 完成定义

本计划完成后，应达到以下状态：

1. 用户在 `nearby` 首次切换时，通常能直接看到前后相邻块进入预览
2. `extended` 相对 `nearby` 有明确的正文增量
3. `core / nearby / extended / full` 四层不再只是状态名变化，而是对应真实不同的正文结果
4. 预算、折叠、导航、全文模式与新局部预览方案可稳定共存
