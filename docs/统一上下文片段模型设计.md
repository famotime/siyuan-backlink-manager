# 统一上下文片段模型设计

## 目的

本文用于执行 `Phase 1 / WP-01`：设计统一的上下文片段模型。

目标是解决当前搜索、筛选、展示三套逻辑依赖不同上下文表达的问题，形成后续开发的统一数据基础。

本文是设计文档，不代表当前代码已实现。

## 设计目标

统一上下文片段模型需要满足以下要求：

1. 能表达当前所有正式上下文来源
2. 能扩展接入相邻块等新来源
3. 能同时服务于搜索、筛选、展示
4. 能承载命中来源、命中摘要、展示层级、预算控制
5. 能以增量方式接入当前实现，而不是要求一次性推翻全部旧结构

## 当前问题

当前实现里：

- 搜索依赖拼接后的文本串
- 筛选依赖定义块 ID、文档 ID 等结构对象
- 展示依赖 Protyle 对文档片段的渲染

这三者有关联，但没有统一的中间表示层。

结果是：

- 命中来源难以解释
- 新增上下文来源要改多处逻辑
- 展示层级难以统一
- 预算控制只能散落在各处实现

## 设计原则

### 1. 统一到“片段”而不是“整串文本”

搜索和展示都应基于一组上下文片段进行，而不是基于单一拼接字符串。

### 2. 片段必须带来源信息

每个片段都必须知道自己来自哪里，否则无法做命中解释、来源标签和预算控制。

### 3. 片段必须带展示层级

上下文不是只有“纳入/不纳入”，而是要区分默认层、扩展层和全文层。

### 4. 搜索、筛选、展示共享片段，但可使用不同字段

三者共享同一片段集合，但各自关注的字段可以不同：

- 搜索看 `searchText`
- 筛选看 `filterRefs`、`sourceType`
- 展示看 `displayText`、`visibilityLevel`

### 5. 先兼容，后替换

第一阶段应允许上下文片段模型与旧结构并存，降低重构风险。

## 核心数据结构

## 一、来源类型枚举

建议新增来源类型枚举：

```ts
type BacklinkContextSourceType =
  | "self"
  | "document"
  | "parent"
  | "child_headline"
  | "child_list"
  | "sibling_prev"
  | "sibling_next"
  | "expanded";
```

说明：

- `self`：反链块自身
- `document`：文档名及文档级辅助信息
- `parent`：父级路径片段
- `child_headline`：标题下子级片段
- `child_list`：列表项下子级片段
- `sibling_prev`：前一个相邻块
- `sibling_next`：后一个相邻块
- `expanded`：扩展结构片段，作为后续兼容扩展保留位

说明补充：

- `expanded` 不是具体结构来源，而是用于承接未来的扩展结构壳
- 若后续需要更细拆分，也可以继续增加枚举值，但不建议第一版过细

## 二、展示层级枚举

建议新增展示层级枚举：

```ts
type BacklinkContextVisibilityLevel =
  | "core"
  | "nearby"
  | "extended"
  | "full";
```

建议定义：

- `core`：默认展示，最小足够理解
- `nearby`：局部扩展层
- `extended`：更大结构范围
- `full`：全文

## 三、命中类型枚举

建议新增命中类型枚举：

```ts
type BacklinkContextMatchType =
  | "text"
  | "anchor"
  | "document"
  | "filter_related"
  | "filter_document";
```

说明：

- `text`：普通文本命中
- `anchor`：锚文本命中
- `document`：文档信息命中
- `filter_related`：因关联定义块筛选而保留
- `filter_document`：因文档筛选而保留

其中 `filter_related` 和 `filter_document` 不是搜索命中，但有助于后续统一“保留原因”表达。

## 四、上下文片段接口

建议核心接口如下：

```ts
interface IBacklinkContextFragment {
  id: string;
  backlinkBlockId: string;
  rootId: string;
  sourceType: BacklinkContextSourceType;
  visibilityLevel: BacklinkContextVisibilityLevel;

  sourceBlockId?: string;
  parentBlockId?: string;
  order?: number;
  depth?: number;

  text: string;
  displayText: string;
  searchText: string;
  anchorText: string;

  refBlockIds: string[];
  includeCurDocDefBlockIds: string[];
  includeRelatedDefBlockIds: string[];

  searchable: boolean;
  filterable: boolean;
  defaultVisible: boolean;

  matched: boolean;
  matchTypes: BacklinkContextMatchType[];
  matchKeywords: string[];
  matchSummary?: string;
}
```

## 字段说明

### 标识字段

- `id`：片段唯一 ID
- `backlinkBlockId`：所属反链块 ID
- `rootId`：所属文档 ID
- `sourceBlockId`：该片段实际对应的块 ID
- `parentBlockId`：用于还原结构关系

### 来源与层级字段

- `sourceType`：片段来源
- `visibilityLevel`：属于哪一层展示
- `defaultVisible`：在默认策略下是否显示

### 排序与结构字段

- `order`：片段在同一反链上下文中的顺序
- `depth`：片段结构深度

### 文本字段

- `text`：原始文本
- `displayText`：用于展示的文本
- `searchText`：用于搜索的规范化文本
- `anchorText`：从该片段中提取的锚文本

### 引用字段

- `refBlockIds`：该片段中所有引用的块 ID
- `includeCurDocDefBlockIds`：该片段中包含的当前文档定义块
- `includeRelatedDefBlockIds`：该片段中包含的关联定义块

### 行为字段

- `searchable`：是否参与搜索
- `filterable`：是否参与筛选

### 命中字段

- `matched`：是否命中
- `matchTypes`：命中类型集合
- `matchKeywords`：命中的关键词
- `matchSummary`：用于 UI 的命中摘要

## 五、反链级聚合接口

建议对单条反链再定义一个统一聚合对象：

```ts
interface IBacklinkContextBundle {
  backlinkBlockId: string;
  rootId: string;

  fragments: IBacklinkContextFragment[];

  visibleFragments: IBacklinkContextFragment[];
  matchedFragments: IBacklinkContextFragment[];

  includeCurDocDefBlockIds: Set<string>;
  includeRelatedDefBlockIds: Set<string>;

  matchSummaryList: string[];
  primaryMatchSourceType?: BacklinkContextSourceType;
}
```

作用：

- 作为搜索、筛选、展示共享的单条反链上下文聚合对象
- 替代当前散落在 `IBacklinkBlockNode` 中的多种上下文字段

## 六、预算接口

为后续预算控制预留统一结构：

```ts
interface IBacklinkContextBudget {
  maxFragments: number;
  maxCharacters: number;
  maxDepth: number;
  maxExpandedNodes: number;
}
```

说明：

- `WP-01` 只设计接口，不实现预算裁剪逻辑
- 真正接入预算控制在 `WP-08`

## 与当前模型的映射关系

## 一、与 IBacklinkBlockNode 的映射

当前 `IBacklinkBlockNode` 中已有字段可以视为片段来源的原始材料：

- `block` -> `self`
- `documentBlock` -> `document`
- `parentMarkdown` -> `parent`
- `headlineChildMarkdown` -> `child_headline`
- `parentListItemTreeNode` / `listItemChildMarkdown` -> `child_list`

映射建议：

- `IBacklinkBlockNode` 在过渡期保留
- 新增 `contextFragments` 字段
- 逐步让旧字段退化为“片段生成输入”

建议过渡结构：

```ts
interface IBacklinkBlockNode {
  ...
  contextFragments?: IBacklinkContextFragment[];
  contextBundle?: IBacklinkContextBundle;
}
```

## 二、与 IBacklinkData 的映射

`IBacklinkData` 当前更偏渲染结果和文档片段引用。  
后续建议：

- 保留其用于 Protyle 渲染的必要字段
- 但展示裁剪、命中摘要、来源标签都应从 `contextBundle` 获取

## 三、与搜索逻辑的映射

当前搜索依赖：

- 文本拼接结果
- 锚文本拼接结果

后续建议：

- 对每个 `fragment` 分别计算命中
- 汇总为 `bundle.matchedFragments`
- 再决定结果是否保留、默认显示哪些片段、摘要显示哪些命中

## 四、与筛选逻辑的映射

当前筛选依赖：

- 当前文档定义块集合
- 关联定义块集合
- 文档集合

后续建议：

- 保留这些筛选维度
- 但来源于 `fragment` 聚合统计，而不是继续散落在多处字段中

## 默认层级映射建议

第一版建议采用以下默认归属：

| 来源类型 | 默认层级 | 默认可见 |
| --- | --- | --- |
| `self` | `core` | 是 |
| `document` | `core` | 是 |
| `parent` | `nearby` | 否 |
| `child_headline` | `nearby` | 否 |
| `child_list` | `nearby` | 否 |
| `sibling_prev` | `nearby` | 否 |
| `sibling_next` | `nearby` | 否 |
| `expanded` | `extended` | 否 |

说明：

- `core` 应保持稳定和低噪音
- `nearby` 是默认的第一层展开目标
- `extended` 是第二层展开目标

## 命中汇总建议

单片段命中后，建议按以下顺序汇总为反链级结论：

1. 记录命中的片段列表
2. 统计命中的来源类型
3. 提取优先级最高的命中作为主命中
4. 为 UI 生成 1 到 3 条命中摘要

主命中优先级建议：

1. `self`
2. `sibling_prev` / `sibling_next`
3. `parent`
4. `child_list` / `child_headline`
5. `document`
6. `expanded`

说明：

- 这是“命中解释优先级”，不是结果排序权重
- 真正排序策略后续可单独设计

## 增量接入方案

建议采用三步接入，而不是一步替换。

### 第一步

在 `IBacklinkBlockNode` 生成完成后，额外生成 `contextFragments`。

此时：

- 旧字段继续保留
- 新片段仅用于验证和调试

### 第二步

让搜索改为基于 `contextFragments` 命中。

此时：

- 旧展示逻辑仍可继续使用
- 但命中来源和摘要已经可以建立

### 第三步

让展示和筛选也逐步切换到 `contextBundle`。

此时：

- 旧字段只保留必要兼容层
- 新增来源、预算控制、展示层级统一走片段模型

## 本设计刻意不做的事

本文不在 `WP-01` 解决以下问题：

- 具体 UI 长什么样
- 展开按钮放哪里
- 预算如何裁剪
- 按文档/按反链模式怎么切换
- 搜索结果最终如何排序

这些分别属于后续工作包：

- `WP-04`
- `WP-05`
- `WP-06`
- `WP-08`
- `WP-10`

## 验收标准

`WP-01` 完成的标准不是代码跑通，而是：

1. 上下文来源、层级、命中、预算都已有统一数据结构
2. 能从当前实现字段映射到新结构
3. 能支持后续 `WP-02` 到 `WP-10` 的持续扩展
4. 增量接入路径明确，不会强制一次性重构

## 当前结论

后续开发应以“反链级聚合对象 + 上下文片段数组”为核心模型推进。  
从这一层开始统一后，搜索、筛选、展示的规则才有可能真正一致。
