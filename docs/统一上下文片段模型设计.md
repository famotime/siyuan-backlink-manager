# 统一上下文片段模型设计

## 文档目的

本文记录当前仓库已经落地的统一模型方向，以及仍然保留的兼容边界。

这不是“纯设计目标稿”，而是当前实现基线文档。

## 当前统一模型

当前代码已经基本形成三层边界：

1. 正文层：`sourceWindow.contextPlan`
2. 说明层：`contextBundle.explanationFragments`
3. 元信息层：`contextBundle.metaInfo`

对应目标是：

- 正文范围只由原文窗口决定
- 说明、命中、预算不再伪装成正文
- 文档标题等辅助信息不再混入四层正文范围

## 一、当前正文模型

当前正文主结构由 `IBacklinkSourceWindow` 承载，实际消费主语义已经转向 `contextPlan`：

```ts
interface IBacklinkSourceWindow {
  rootId: string;
  anchorBlockId: string;
  startBlockId: string;
  endBlockId: string;
  focusBlockId: string;
  windowBlockIds: string[];
  visibleBlockIds?: string[];
  orderedVisibleBlockIds?: string[];
  contextPlan?: {
    identity: {
      rootId: string;
      anchorBlockId: string;
      focusBlockId: string;
      sourceDocumentOrder?: number;
    };
    bodyRange: {
      startBlockId: string;
      endBlockId: string;
      windowBlockIds: string[];
    };
    orderedVisibleBlockIds: string[];
    collapsedBlockIds: string[];
    structuralShellBlockIds: string[];
  };
  defaultExpandMode: string;
}
```

当前语义：

- `bodyRange`：Protyle 需要加载的最小原文窗口
- `orderedVisibleBlockIds`：当前层级默认可见的正文块序列
- `collapsedBlockIds`：属于窗口但默认折叠/占位的块
- `structuralShellBlockIds`：用于定位结构的壳层块

当前状态：

- getter 已能统一读取 `contextPlan` 与 legacy root 字段
- 调用侧已开始只消费 getter 结果

## 二、当前说明片段模型

当前 `IBacklinkContextBundle` 已经形成“说明片段 + 元信息”的双层结构：

```ts
interface IBacklinkContextBundle {
  backlinkBlockId: string;
  rootId: string;
  fragments: IBacklinkContextFragment[];
  explanationFragments: IBacklinkContextFragment[];
  visibleFragments: IBacklinkContextFragment[]; // 兼容镜像
  metaInfo: IBacklinkContextMetaInfo;
  matchedFragments: IBacklinkContextFragment[];
  includeCurDocDefBlockIds: Set<string>;
  includeRelatedDefBlockIds: Set<string>;
  matchSummaryList: string[];
  primaryMatchSourceType?: BacklinkContextSourceType;
  budgetSummary?: IBacklinkContextBudgetSummary;
}
```

当前语义：

- `fragments`：全量来源片段
- `explanationFragments`：当前层级用于解释“为什么命中 / 为什么显示”的片段
- `visibleFragments`：旧调用方兼容镜像，主语义已退居次要
- `metaInfo`：不计入正文范围的辅助信息

## 三、当前元信息模型

目前已正式拆出的元信息主要是文档标题：

```ts
interface IBacklinkContextMetaInfo {
  documentTitle?: {
    key: "documentTitle";
    text: string;
    renderMarkdown?: string;
    searchText: string;
    visibilityRole: "meta";
    searchable: boolean;
    matched: boolean;
    matchTypes: string[];
    matchKeywords: string[];
  };
  matchedFieldKeys: string[];
  matchSummaryList: string[];
  primaryMatchKey?: string;
}
```

当前状态：

- 文档标题命中已不再伪装成正文片段命中
- 文档标题说明已通过 `metaInfo` 注入摘要层

## 四、当前来源类型

当前来源枚举仍保留旧来源桶，但已完成“正文和元信息不混用”的关键分层：

```ts
type BacklinkContextSourceType =
  | "self"
  | "document"
  | "parent"
  | "child_headline"
  | "child_list"
  | "sibling_prev"
  | "sibling_next"
  | "expanded";
```

当前口径：

- `document`：仍保留为来源类型，但默认不可见、不可搜索、不可筛选
- `expanded`：仍作为兼容来源存在，不等价于正文的 `extended` 窗口

这意味着：

- 来源模型已经能支撑解释层
- 但还没有完全等价为“单一 planner 结果”

## 五、当前已实现的关键对齐

### 1. 元信息与正文已拆层

- `documentTitle` 已落到 `metaInfo`
- `document` 不再是默认正文片段

### 2. 正文排序主链路已收敛

- 正文主路径优先 `sourceWindow/contextPlan`
- `orderedVisibleBlockIds` 已成为正文可见序列主证据
- DOM 过滤层已优先消费这条线性序列

### 3. 列表规则已对齐正式窗口语义

- 列表 `core` 已收紧为“壳 + 焦点块”
- 列表 `nearby` 默认只保留邻居壳层与首层直接子项

### 4. 降级态已显式化

- source window 缺失时，头部控制区会显示“原文上下文不可用，当前显示为降级结果”
- 这条提示属于异常态说明，不属于正文内容

## 六、仍保留的兼容边界

### 1. `visibleFragments` 仍未删除

现状：

- 仍作为 `explanationFragments` 的兼容镜像保留

### 2. `sourceWindow` 根字段仍未完全删除

现状：

- getter 已开始统一读取
- 但类型层和部分兼容路径仍保留 root 字段

### 3. preview assembly 仍未彻底下线

现状：

- 已退出正文主路径
- 仍作为兼容模块与测试资产存在

## 七、推荐后的最终目标

当前模型的合理终点应是：

```ts
interface IBacklinkContextPlan {
  identity: {
    rootId: string;
    anchorBlockId: string;
    focusBlockId: string;
    sourceDocumentOrder?: number;
  };
  bodyRange: {
    startBlockId: string;
    endBlockId: string;
    windowBlockIds: string[];
  };
  orderedVisibleBlockIds: string[];
  collapsedBlockIds: string[];
  structuralShellBlockIds: string[];
}
```

以及：

- 正文只消费 `IBacklinkContextPlan`
- 说明只消费 `explanationFragments + metaInfo`
- legacy root 字段和 preview 拼装逻辑只存在于 adapter / fallback 层

## 当前结论

统一模型已经不再停留在“设计目标”，而是已经在仓库里形成了可用主骨架：

- `contextPlan` 负责正文窗口
- `explanationFragments` 负责说明
- `metaInfo` 负责辅助信息

后续真正还需要完成的，是把兼容字段和兼容模块继续向外压缩，而不是再回到“所有逻辑共用一串拼接文本”的旧结构。
