# 近邻状态显示问题分析

## 文档定位

本文基于正式规则分析 `nearby` 的当前实现缺口。

正式规则见：

- `docs/上下文状态渐进显示规则.md`

## 正式规则中的 `nearby`

`nearby` 的目标不是“把更多内容带出来”，而是补足理解当前引用所需的局部前后文。

核心要求：

- 普通块：当前锚点单元加前后相邻单元
- 列表场景：当前列表项结构壳加前后同级列表项
- 相邻范围优先保持在同一节内
- 邻居父项默认展示结构壳与首层直接子项，不默认展开整棵深层子树

## 当前实现现状

当前 `nearby` 主要由 `src/service/backlink/backlink-source-window.js` 中的 `buildNearbyBacklinkSourceWindow` 计算。

现状分两类：

- 普通块：以文档顺序前后块作为近邻窗口
- 列表场景：以 `previousSiblingBlockId` / `nextSiblingBlockId` 指向的前后列表项作为窗口边界，并把边界列表项的完整子树纳入

## 主要问题

### 1. 没有限制在同一所属标题节内

当前普通块近邻按文档顺序取前后块，没有额外判断是否跨出当前所属标题节。

结果：

- 在多级标题文档中，`nearby` 可能跨节
- 用户看到的不是“局部近邻”，而是“线性邻居”

### 2. 列表邻居范围过大

当前列表近邻会把相邻列表项的完整子树直接纳入窗口。

结果：

- 一旦邻居项下存在多层子项，`nearby` 会迅速膨胀
- 首次展开就可能塞入大量无关内容
- 近邻层失去“局部补充”的作用

### 3. 缺少“结构壳 + 深层折叠”表达能力

正式规则要求：

- 用户能看见相邻列表项属于什么主题
- 但不要求默认看见邻居的所有深层内容

当前 source window 更擅长表达“完整块范围”，不擅长表达“结构壳可见、深层后代折叠”的半展开状态。

### 4. source window 与 preview 组装规则仍不完全一致

局部预览既可能走 source window 裁剪，也可能走 preview markdown 组装。

这导致：

- 相同的 `nearby` 规则在不同结构场景下呈现方式不同
- 后续要补结构壳能力时，必须同时收敛两条渲染路径

## 影响模块

P0 直接模块：

- `src/service/backlink/backlink-source-window.js`
- `src/service/backlink/backlink-preview-assembly.js`
- `src/components/panel/backlink-document-interaction.js`
- `src/components/panel/backlink-protyle-rendering.js`

P1 关联模块：

- `src/service/backlink/backlink-context-rules.js`
- `src/components/panel/backlink-panel-header.js`
- `src/components/panel/backlink-document-view-state.js`

## 建议修正方向

1. 先把 `nearby` 的边界收敛到“同一节内、同级优先”。
2. 列表邻居改成“结构壳优先，深层后代默认折叠”。
3. 给 source window 或 preview fragment 增加“结构壳渲染契约”，避免只能在“单块”和“整棵子树”之间二选一。
4. 为普通块、多级标题、父项邻居、深层列表分别补充测试。

## 结论

当前 `nearby` 已经能表达“前后邻居存在”，但还不能稳定表达正式规则中的“局部、同节、可控范围”。

这不是单点 bug，而是范围判定与渲染契约需要一起调整。
