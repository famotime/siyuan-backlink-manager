# 近邻状态显示问题分析

## 问题描述

示例1场景（反链在普通文本块）未能按规则显示"近邻"状态的上下文范围。

## 规则要求

**近邻**：反链所在段落加上前后段落，或者反链所在列表项加上前后列表项，如果前后列表项包含父节点则涵盖整个父节点内容。

## 问题根因

在 `src/service/backlink/backlink-source-window.js` 的 `buildNearbyBacklinkSourceWindow` 函数（第321-343行）中：

```javascript
function buildNearbyBacklinkSourceWindow(backlinkBlockNode, context) {
  const listItemAnchorBlockId = resolveListItemAnchorBlockId(backlinkBlockNode, context);
  const anchorBlockId = listItemAnchorBlockId || backlinkBlockNode.block.id;
  const anchorStartIndex = context.indexById.get(anchorBlockId);

  const startBlockId = backlinkBlockNode.previousSiblingBlockId || anchorBlockId;
  const endBlockId = backlinkBlockNode.nextSiblingBlockId || anchorBlockId;
  const startIndex = context.indexById.get(startBlockId) ?? anchorStartIndex;
  const endIndex = getBlockRangeEndIndex(endBlockId, context) ?? getBlockRangeEndIndex(anchorBlockId, context);

  return buildSourceWindowFromRange({...});
}
```

### 核心问题

1. **依赖 sibling 数据不准确**：
   - `previousSiblingBlockId` 和 `nextSiblingBlockId` 来自 `collectSiblingBlocks` 收集的数据
   - 这些数据基于**同一父节点下的兄弟块**（通过 SQL 查询 `parent_id` 相同的块）
   - 但"近邻"的语义是**文档顺序上的前后相邻块**，不一定是同一父节点

2. **示例场景分析**：
   - 反链块：`20251116213835-066dqds`（段落，parent_id: `20251116213835-iorhvx7`）
   - 前一个块：`20251116213835-dr4a0fg`（段落，parent_id: `20251116213835-iorhvx7`）
   - 后一个块：`20251116213835-2eb5dgk`（列表，parent_id: `20251116213835-iorhvx7`）

   这个场景下，三个块确实是同一父节点的兄弟，但如果：
   - 前一个块是 `20251116213835-gi8z7o4`（引用块，parent_id: `20251116213835-iorhvx7`）
   - 反链块是 `20251116213835-066dqds`（段落，parent_id: `20251116213835-iorhvx7`）
   - 后一个块是 `20251116213835-2eb5dgk`（列表，parent_id: `20251116213835-iorhvx7`）

   那么 sibling 查询可能会遗漏某些块。

3. **真正的问题**：
   - 需要基于 `orderedDocumentBlocks` 的索引来找前后相邻块
   - 而不是依赖预先收集的 `previousSiblingBlockId` / `nextSiblingBlockId`

## 修复方案

修改 `buildNearbyBacklinkSourceWindow` 函数，当 `previousSiblingBlockId` 和 `nextSiblingBlockId` 都不存在时，回退到从 `orderedDocumentBlocks` 中查找前后相邻块：

```javascript
function buildNearbyBacklinkSourceWindow(backlinkBlockNode, context) {
  const listItemAnchorBlockId = resolveListItemAnchorBlockId(backlinkBlockNode, context);
  const anchorBlockId = listItemAnchorBlockId || backlinkBlockNode.block.id;
  const anchorStartIndex = context.indexById.get(anchorBlockId);
  if (anchorStartIndex === undefined) {
    return null;
  }

  let startBlockId = backlinkBlockNode.previousSiblingBlockId;
  let endBlockId = backlinkBlockNode.nextSiblingBlockId;

  // 当没有预先收集的 sibling 数据时，从文档顺序中查找
  if (!startBlockId && !endBlockId) {
    const prevIndex = anchorStartIndex - 1;
    const nextIndex = anchorStartIndex + 1;
    startBlockId = prevIndex >= 0 ? context.orderedDocumentBlocks[prevIndex]?.id : null;
    endBlockId = nextIndex < context.orderedDocumentBlocks.length ? context.orderedDocumentBlocks[nextIndex]?.id : null;
  }

  startBlockId = startBlockId || anchorBlockId;
  endBlockId = endBlockId || anchorBlockId;

  const renderStartBlockId = resolveFirstDescendantBlockId(startBlockId, context);
  const startIndex = context.indexById.get(startBlockId) ?? anchorStartIndex;
  const endIndex = getBlockRangeEndIndex(endBlockId, context) ?? getBlockRangeEndIndex(anchorBlockId, context);

  return buildSourceWindowFromRange({
    backlinkBlockNode,
    context,
    anchorBlockId,
    startIndex,
    endIndex,
    startBlockId: renderStartBlockId,
  });
}
```

## 修复效果

- 保留了列表项场景的原有逻辑（使用 `previousSiblingBlockId` / `nextSiblingBlockId`）
- 为普通段落块场景提供了回退机制（从文档顺序查找相邻块）
- 所有测试用例通过（8/8）
